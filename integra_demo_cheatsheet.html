Tech Stack -- Why These Choices
FastAPI + Railway
FastAPI is async-native, which matters for chaining multiple inference calls in a single request. Compared to Flask, it has built-in request validation via Pydantic and auto-generates OpenAPI docs.

Railway was chosen over HuggingFace Spaces for the backend because it runs a standard Docker container with no framework lock-in, auto-deploys from GitHub, and exposes a stable HTTPS endpoint. HF Spaces have cold-start issues and are better suited to demo-only Gradio apps.
React + Vite + Vercel
Streamlit was the original frontend. It was replaced because Streamlit re-runs the entire script on every interaction, making custom UI state (bottom nav active states, animated loading, multi-mode journal page) very difficult to control.

React gives component-level state, smooth transitions, and full control over mobile layout. Vite provides near-instant dev server HMR. Vercel is purpose-built for React/Vite: zero config, instant previews, env var management, and vercel.json rewrites for client-side routing.
HuggingFace Inference API
Running RoBERTa and BART locally requires downloading ~500MB of model weights, which breaks Railway deployments and inflates Docker image size.

The Inference API offloads model hosting entirely. The tradeoff is cold-start latency on the first call of a session (~5-8 seconds). This is acknowledged as a known limitation. In a production deployment these would be warm, dedicated endpoints.
LangChain + ChromaDB
LangChain manages the retrieval chain: embedding the user query, running cosine similarity search against ChromaDB, formatting retrieved chunks, and injecting them into the GPT-4o context window.

ChromaDB was chosen over Pinecone because it runs entirely locally (no external service dependency), embeds in the Docker container, and is well-supported by LangChain. For 4,450 vectors it performs well with no need for managed infrastructure.
GPT-4o for RAG
GPT-4o handles the final generation step after retrieval. It was chosen over smaller open models because the task requires nuanced, emotionally calibrated language grounded in clinical literature -- not just factual retrieval.

The system prompt enforces IFS and ACT-aligned response style (reflection, open questions, no clinical advice), which requires a model with strong instruction-following.
Recharts for Visualization
Recharts is a React-native charting library built on D3. It was chosen over plain D3 because it integrates cleanly with React's component model and state, supports responsive containers natively, and requires far less boilerplate for line and bar charts.

The emotion timeline line chart uses LineChart with one Line per emotion. The theme frequency chart uses BarChart with a horizontal layout.
RoBERTa Emotion Remapping -- Full Weight Matrix
The base model (j-hartmann/emotion-english-distilroberta-base) outputs 7 labels. A custom weighting layer converts these to 8 integration-specific labels. Each base score is multiplied by the weight and summed across all base emotions that contribute to that target.
RoBERTa Base Label Maps to Integra Label(s) Weight(s) Rationale
surprise awe overwhelm 0.8 / 0.3 Psychedelic experiences consistently report awe and surprise as synonymous. Overwhelm is a secondary component of high-valence surprise states.
sadness grief confusion 0.9 / 0.2 Grief is the integration-relevant label for sadness. Confusion co-occurs with grief in integration literature -- processing loss often surfaces disorientation.
joy gratitude calm joy 0.5 / 0.3 / 0.4 RoBERTa has no gratitude label. Gratitude-dominant integration text routes through joy. Calm and joy retained at lower weights. Known limitation: gratitude-dominant entries may show joy as dominant in output.
fear fear overwhelm 0.9 / 0.4 Fear maps directly. Overwhelm is a frequent co-occurrence in psychedelic contexts -- fear during a session often presents as overwhelm in written retrospective accounts.
anger overwhelm confusion 0.5 / 0.4 Anger is rare in integration journaling and not a first-class label. In integration contexts it typically surfaces as overwhelm or confusion about emotional reactions.
disgust confusion fear 0.5 / 0.3 Disgust is also rare in this domain. When present in integration writing it tends to reflect confronting ego material -- closer to confusion or fear than to its base label.
neutral calm 0.6 Neutral text in integration journaling often reflects a settled, observational state. Calm is the closest integration-specific equivalent. Low weight prevents over-attribution.
Known limitation: Emotions below a 0.2 confidence threshold are filtered from output. Because RoBERTa has no dedicated gratitude label, entries where gratitude is the primary tone may return joy as the top label. This is flagged in the Limitations slide and in the demo script.
FDA-Approved Mental Health Chatbots -- Research Context
Current FDA Landscape (as of early 2026)
The FDA regulates digital mental health tools under its Software as a Medical Device (SaMD) framework. As of the capstone date, no AI-powered chatbot has received full FDA approval as a standalone mental health treatment.

The FDA has granted Breakthrough Device Designation to some digital therapeutics (DTx), notably Freespira (PTSD/panic disorder, cleared 2019 as a Class II device) and a handful of CBT-based apps. These are not chatbots -- they are structured digital therapy programs.

Woebot (CBT-based chatbot) received a Breakthrough Device Designation from the FDA in 2021 for a specific adolescent depression indication, but as of 2025 had not completed the full de novo pathway to clearance.
Why This Matters for Integra
Integra is explicitly not positioned as a medical device or therapy tool. This is a deliberate design and legal choice.

The disclaimer chip ("Indy is an AI companion, not a therapist") on the login screen and in the chat interface is the front-line signal of this positioning.

The system prompt reinforces this: Indy reflects and asks questions grounded in published integration protocols. It does not diagnose, prescribe, or provide clinical advice. This mirrors the operational model of apps like Woebot's general wellness tier, which fall outside FDA regulation as a "general wellness" product rather than a medical device.

RAG grounding is a safety feature in this context: anchoring responses to published MAPS and Yale protocols reduces hallucination risk and keeps responses within the bounds of established integration practice.
Woebot (Woebot Health)
CBT-based conversational agent. FDA Breakthrough Device Designation (2021) for adolescent depression. Peer-reviewed trials published in JMIR Mental Health (Fitzpatrick et al., 2017). Operates in rule-based + NLP hybrid mode, not a generative LLM. General wellness tier (no FDA regulation) for adult use.
Wysa (Wysa Inc.)
CBT and DBT-informed chatbot. CE-marked as a Class I medical device in the EU (2023). FDA Breakthrough Device Designation for a major depressive disorder indication. Clinical validation studies published in JMIR mHealth. Uses scripted conversation trees with LLM-assisted transitions.
Freespira (Palo Alto Health Sciences)
FDA 510(k) cleared Class II medical device for PTSD and panic disorder. Not a chatbot -- it is a biofeedback-based respiratory training app. Relevant because it is the clearest example of an FDA-cleared digital therapeutic in the PTSD/trauma space that overlaps with Integra's user population.
Research sources: JMIR Mental Health (2024, 2025 systematic reviews on LLM-based mental health chatbots), FDA SaMD guidance documents (2022, 2023 updates), PMC review "Safety and Efficacy of AI Chatbots for Mental Health" (medRxiv 2024), Woebot Health white papers.
Model Decision Rationale -- Quick Reference
RoBERTa for Emotion
Distilled model, fast inference, well-benchmarked on emotional text. The Hartmann variant is fine-tuned specifically on emotional language. Alternative (GoEmotions BERT) covers 28 labels -- too granular, no integration-specific mapping available.
HF Inference API ~150ms warm
BART-large-mnli for Themes
Zero-shot classification via natural language hypothesis testing. No training data required -- themes are just strings passed as candidate labels. The keyword layer runs first to avoid cold-start latency on every request.
Zero-shot Fallback safe
text-embedding-3-small
OpenAI's smallest embedding model, optimized for retrieval tasks. 1536 dimensions. Chosen over ada-002 (now deprecated) and text-embedding-3-large (overkill for 4,450 vectors, higher cost per embedding).
4,450 vectors ChromaDB
GPT-4o for Generation
Strong instruction-following needed for the IFS/ACT-aligned system prompt. GPT-4o mini was tested but showed weaker adherence to the reflective response style. GPT-4o cost is acceptable for a demo-scale deployment.
RAG grounded IFS + ACT prompt
Common Q and A -- If Asked During Demo
Why not fine-tune RoBERTa on integration text?
Fine-tuning requires labeled data -- annotated integration journal entries with ground-truth emotion labels. No such dataset exists publicly. The remapping layer is a pragmatic alternative: it uses domain knowledge about psychedelic integration to adjust the base model's output distribution without requiring training data.
How does crisis detection work exactly?
Two layers run in sequence. Layer 1: exact keyword matching against a curated list of crisis phrases (runs in microseconds). Layer 2: cosine similarity between the user message embedding and a set of crisis reference sentences using the same embedding model as the RAG pipeline. If either layer triggers, the pipeline returns the 988 message and skips GPT-4o entirely.
Is Jordan's data real?
No. Jordan's 14 entries are seeded data generated to reflect a realistic integration arc based on patterns described in the MAPS MDMA therapy manual and Yale psilocybin protocol. The emotional arc (awe/overwhelm resolving toward gratitude/joy) is consistent with outcomes reported in the clinical literature. This is disclosed on the Limitations slide.
Why does the theme extractor use two layers?
BART-large-mnli has a cold-start delay of 5-8 seconds on HuggingFace Inference API. Running zero-shot on every entry would make the journal submission feel slow. The keyword layer catches the majority of themes instantly. Zero-shot only runs on the remainder, keeping median latency low while maintaining coverage for plain-language expressions the keyword layer misses.
